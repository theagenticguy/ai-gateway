## Executive Summary

The AI Gateway is a lightweight LLM API proxy built on **Portkey OSS** (v1.15.2), deployed as a containerized service on **AWS ECS Fargate** behind an **Application Load Balancer**. Infrastructure is managed with **Terraform** using battle-tested `terraform-aws-modules` (VPC, ECS, ALB). The container image uses `python:3.13-slim` with multi-stage hardening (non-root, tini, minimal packages), scanned by a 3-phase CI security pipeline (hadolint + checkov pre-build, trivy post-build, cosign signing pre-push). Observability flows through an ADOT sidecar exporting to CloudWatch. Provider API keys live in AWS Secrets Manager, injected at task startup. Total infrastructure cost: ~$215-280/month at medium scale.

## Architecture

### System Architecture

```
Client (SDK / curl / application)
    |
    | HTTPS (port 443)
    v
[AWS WAF v2] --> [Application Load Balancer]
                        |
                        | HTTP (port 8787)
                        v
            [ECS Fargate Task (Private Subnet)]
            +----------------------------------+
            | Portkey Gateway    | ADOT Sidecar |
            | (port 8787)       | (port 4317)  |
            |                   |              |
            | x-portkey-provider|  --> CloudWatch
            | x-portkey-config  |  --> X-Ray
            +----------------------------------+
                  |           |
                  |           +---> Secrets Manager (API keys at startup)
                  |
                  | HTTPS (outbound via NAT GW)
                  v
        [LLM Providers]
        - OpenAI
        - Anthropic
        - AWS Bedrock (via task role, no NAT)
        - Google Vertex AI
        - Azure OpenAI
```

### VPC Layout

```
VPC (10.0.0.0/16)
├── Public Subnet AZ-1 (10.0.1.0/24): ALB, NAT Gateway
├── Public Subnet AZ-2 (10.0.2.0/24): ALB
├── Private Subnet AZ-1 (10.0.10.0/24): Fargate tasks
├── Private Subnet AZ-2 (10.0.20.0/24): Fargate tasks
└── VPC Endpoints: ECR API, ECR DKR, S3 (GW), CloudWatch Logs, Secrets Manager
```

### CI/CD Security Pipeline

```
Phase 1: PRE-BUILD (on every PR)
  hadolint ──> Dockerfile best practices
  checkov ──> Terraform policy enforcement

Phase 2: POST-BUILD (after docker build)
  trivy ────> Image vulns + misconfig + secrets
  syft ─────> SBOM generation (CycloneDX)

Phase 3: POST-SCAN (before push to ECR)
  cosign ──> Keyless image signing (Sigstore OIDC)
```

## Trade-offs and Risks

| Trade-off | Impact | Mitigation |
|---|---|---|
| python:3.13-slim over Chainguard | Higher CVE count in base image | Trivy scanning in CI; upgrade path to Chainguard paid when ready |
| Single NAT Gateway (cost savings) | Outbound LLM calls fail during NAT GW AZ failure | Fargate tasks HA across AZs; only outbound affected; Bedrock uses VPC endpoint |
| No database | No chat history, session persistence | Add Aurora Serverless v2 later if needed; Portkey OSS is stateless by design |
| Portkey OSS no /v1/health | Health checks use GET / ("AI Gateway says hey!") | ALB matcher on HTTP 200; not ideal but functional |
| Terraform BSL license | Restricts commercial competitors (not us) | No impact for internal use; OpenTofu available as fallback |

### Known Limitations

- Portkey OSS lacks enterprise features (SSO, audit logs, advanced guardrails)
- No Redis for response caching — add ElastiCache later if needed
- OTEL instrumentation requires adding SDK to gateway (auto-instrumentation available for Node.js)
- Secret rotation requires ECS redeployment (no hot-reload)
